# V-01 Acceptance — Small Real-User Validation

## 结论

- Status: Ready
- Date opened: 2026-06-25
- Goal: observe 1-3 non-builder users on Production and record whether they can complete ForgeNote's M1 loop without Codex or Owner driving the product decisions.
- Environment target: Production `https://forge-note-gold.vercel.app`
- Scope: validation only; no product code, prompt, schema, RLS, visual rendering, asset library, Stripe, runtime i18n, or automatic learning.

## Gate 1 — Product Intent

- User: a non-builder creator or creator-adjacent user with a real carousel / graphic post idea.
- Task: log in, enter a fuzzy idea, understand or edit one assumption, generate a content plan, decide whether to save it as a recipe, open the recipe detail, and rerun it with a new input.
- Changed path being tested: the full M1 loop that now exists after I-20, I-22, and I-23.
- Loss if not done: ForgeNote keeps accumulating features without knowing whether a real user can understand the loop or sees enough value to save and rerun.
- Out of scope: feature requests during the session; record them, do not implement them in V-01.

## Test Script

Read only this short framing to the user:

```text
请用你自己的一个真实内容想法试 ForgeNote。
目标不是评价你写得好不好，而是看这个工具能不能帮你从模糊想法走到可复用的内容方案。
过程中请尽量自己操作；卡住时直接说你卡在哪里。
```

Do not explain hidden product concepts such as schema, recipe internals, metrics, or tickets.

## Observer Rules

- Do not choose the topic for the user.
- Do not tell the user which assumption to edit.
- Do not tell the user whether the generated content is "good enough".
- Help only with account/login infrastructure if blocked.
- Record exact blockers and visible page states.

## Gate 4 — Metrics Readout

After each session batch, run:

```bash
npm run doctor
npm run metrics
```

Record the metric snapshot here:

```text
Metric readout date:
- activation:
- assumption_edit:
- recipe_save:
- recipe_rerun:
- return_session:
- performance_fill:
- Command result:
- Blocker if metrics unavailable:
```

## Completion Criteria

- At least 1 non-builder user completes or clearly fails at a named step in the Production path.
- The record includes environment, user type, topic, step outcomes, session/recipe ids when available, and the observer conclusion.
- Metrics are read out or the exact metrics blocker is recorded.
- Final status is one of:
  - Pass: at least one user completes first generation, save, and rerun without product-critical confusion.
  - Conditional Pass: the core path works but a concrete friction point must become the next ticket.
  - Fail / Blocked: login, generation, save, rerun, or metrics cannot be completed.
