# I-09 Acceptance — 配方库列表、搜索、筛选、删除

> 票：受保护 `/recipes` 页面展示当前登录用户可见 recipes，支持按名称搜索、按 `intent_type` 筛选、单条删除。
> **状态：Done。** 自动验证、匿名权限边界、真实 Chrome 登录态列表/搜索/筛选/删除均已通过。

## 范围内

- 新增受保护 `/recipes` 页面。
- Server Component 直接使用当前 Auth cookie 绑定的 Supabase client 读取 `recipes`，依赖 RLS 隔离当前用户数据。
- 列表展示：名称、`intent_type`、摘要字段、`created_at`、`updated_at`、`usage_count`、`source_session_id`。
- 搜索：URL 参数 `q`，按 `name` 做 `ilike`。
- 筛选：URL 参数 `intentType`，仅允许 M1 四个 canonical intent。
- 删除：新增 `DELETE /api/recipes/:id`，软删除当前用户可见且未删除的单条 recipe。
- UI 状态：空态、筛选加载态、数据库错误态、删除确认、删除中、删除成功反馈。

## 范围外（硬限制）

- 不做 `/recipes/[id]` 详情页。
- 不做换输入重跑。
- 不做高表现排序 / `perf_score`。
- 不做 Profile / 偏好记忆。
- 不做 F-16 表现回填。
- 不改模型 prompt。
- 不改 I-08 保存逻辑。

## 实现要点

- `/recipes` 鉴权在 Server Component 顶层执行：未登录或 Supabase 未配置 → `redirect("/login")`。
- 页面读取 recipes 时统一过滤 `deleted_at is null`，并按 `created_at desc` 排序。
- 只按 `name` 搜索，未扩展到 `fields` 文本，避免过度设计。
- 列表读取不新增 GET API；页面直接读库，符合“能安全读取就少建 API”要求。
- 删除使用 API Route，保持现有 `/api/forge`、`/api/recipes` 的 JSON 错误码风格。
- `DELETE /api/recipes/:id` 校验顺序：鉴权 → UUID 校验 → RLS + `user_id` + `deleted_at is null` 软删除。
- 他人 / 不存在 / 已删除 recipe 均返回 `RECIPE_NOT_FOUND`，不泄露存在性。
- 软删除写 `deleted_at`，由 DB trigger 维护 `updated_at`。

## 登录态 Chrome 验收

- 入口：Chrome 现有登录态访问 `http://localhost:3000/recipes`，页面停留在 `/recipes`，导航显示当前用户 email。
- 初始列表：显示 2 条 I-08 保存的重复配方，字段包含名称、类型、摘要、创建时间、更新时间、使用次数、来源 session。
- 搜索：输入“备用金”并搜索 → URL 更新为 `/recipes?q=...`，仍显示 2 条命中记录。
- 类型筛选：选择“内容包” → URL 更新 `intentType=content_package`，仍显示 2 条 `内容包`。
