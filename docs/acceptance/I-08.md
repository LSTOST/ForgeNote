## 范围内

- `RecipePanel` 成功态启用「保存配方」+ 轻量命名 UI + 保存中/已保存/失败反馈。
- `ForgeWorkbench` 把 `sessionId` 透传给 `RecipePanel`。

## 范围外（硬限制）

- 不做 `/recipes` 列表页、配方详情、换输入重跑。
- 不做偏好记忆 / `profile_preferences`、不做 F-16。
- 不改模型 prompt、不改生成逻辑。

## 实现要点

- 错误码与 HTTP：`AUTH_REQUIRED` 401 / `VALIDATION_FAILED` 400 / `SESSION_NOT_FOUND` 404 / `DATABASE_ERROR` 500（对齐 API-CONTRACT §3 / §5.5）。
- 校验顺序：解析 JSON → body schema → **鉴权** → name trim 非空 → 读 session → 有无 `recipe_snapshot` → 插入。鉴权早于业务校验（与 `/api/forge` 一致）。
- session 读取经 RLS（`auth.uid() = user_id`）：他人/不存在的 `sessionId` 都查不到 → `SESSION_NOT_FOUND`，不泄露存在性。
- `recipes` 写入映射：`user_id=user.id`、`name=trimmed`、`intent_type=session.intent_type`、`schema={intentType,version:1}`、`fields=recipe_snapshot`、`acceptance/variables/negative_rules` 从快照拆出、`source_session_id=session.id`。
- 同名允许：每次保存都是独立 INSERT，不 upsert、不覆盖。
- 前端：成功态显示「保存配方」；命名 UI 默认值用 `recipe.name`；名称为空禁用保存；保存中 loading；成功显示「已保存到配方库」；失败显示 inline error 且不清空已生成结果。切换 session 时以 `key={sessionId}` 重挂面板重置保存态。

## 登录态验收（尚未复验 — 阻断 Done）

以下用例需真实 Supabase 登录 cookie，本次执行环境无可用登录态，**全部未运行**，仅由代码走查覆盖。在它们逐项通过前，I-08 维持 Review，不判定 Done：

- [ ] 登录后保存当前生成 session → `recipes` 新增一条，`source_session_id` / `user_id` / `fields` 正确写入。

- [ ] 登录后空 name → 400 `VALIDATION_FAILED`。
- [ ] UI：保存成功显示「已保存到配方库」；失败显示 inline error 且结果不丢。
