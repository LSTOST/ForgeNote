# I-21 Acceptance — Forge Session URL Continuity

## 结论

- Status: Done
- Date: 2026-06-24
- Scope implemented: after `/api/forge` returns a completed session, `/forge` updates the browser URL to `/forge?session=<id>`; after a draft-producing generation failure, `/forge` also updates to the draft session URL; new-task actions clear stale `?session=`.
- Why this ticket exists: I-20 proved the onboarding-first generation path, but a plain refresh after generation could lose the result because the URL stayed `/forge`. That violates the M1 user-path baseline: “刷新页面后仍能找到结果”.

## Gate 1 — Product Intent

- User: logged-in creator using `/forge` to turn a rough idea into a reusable content plan.
- Task: generate once, then refresh or return to the URL without losing the result.
- Changed path: `/forge` generation now creates an addressable result URL immediately.
- Loss if not done: the product claims session persistence, but the first generated result is not reliably recoverable by normal browser behavior.
- Out of scope: session history, version list, asset library, visual recipe, content package redesign, prompt/schema/RLS changes.

## Implementation

- `src/components/forge/ForgeWorkbench.tsx`
  - imports `useRouter`.
  - on generation success, reads `data.sessionId` and calls `router.replace("/forge?session=<id>", { scroll: false })`.
  - on generation failure with `draft.sessionId`, preserves error state and also writes `/forge?session=<id>`.
  - `prepareDirection()` and `handleNew()` clear stale session query by replacing the URL with `/forge`.

No API, DB, RLS, prompt, or schema changes.
