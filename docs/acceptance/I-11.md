> 票：受保护 `/profile` 管理当前用户的偏好假设（profile_preferences）；`/forge` 下次生成把偏好作为 `source="profile"` 假设带出；可从 Forge 把已编辑的假设「记住为偏好」。

## 范围内

- 受保护 `/profile`（Server Component 鉴权，未登录 → `/login`）。
- 偏好 CRUD：
  - `GET /api/profile/preferences`（列出当前用户偏好）
  - `POST /api/profile/preferences`（创建/更新，按唯一键 `(user_id, intent_type, dimension_key)` upsert）
  - `PUT /api/profile/preferences/:id`（改 value）
  - `DELETE /api/profile/preferences/:id`（删除单条）
- `/profile` UI：新增表单 / 行内编辑值 / 删除防误触 / 空态 / 刷新态 / 保存中 / 成功反馈 / 错误态。
- edited assumptions 写入：`/forge` AssumptionPanel 每条假设加「记住为偏好」（★）按钮 → upsert profile_preferences。
- 下次带出：`/forge` Server Component 按 RLS 读取 `content_package` 偏好 → 映射为 `source="profile"` 假设 → 作为 ForgeWorkbench 初始假设种子；生成时随假设提交并落入 `session.assumptions`；AssumptionPanel 显示「来自偏好」。

## 范围外（硬限制）

- 不做自动学习 / 不从历史 session 自动总结偏好。
- 不做 F-16 / I-12 表现回填、不做 I-17 i18n 资源外化、不做 UI 多语言切换。
- 不改 `content_package` / `output_locale` / 模型 prompt / RLS。
- 不引入新依赖。

## 数据层

- **无 migration**：`profile_preferences` 表与 RLS（`auth.uid() = user_id`，唯一键 `(user_id, intent_type, dimension_key)`）在 `0001_init.sql` 已存在。
- 偏好以「假设维度」存储：`intent_type` + `dimension_key` + `dimension_label` + `value` + `source`（M1 固定 `manual`）。

## 偏好注入设计（关键决策）

- 注入点选在 **`/forge` Server Component**（DAL 模式，与现有 session 预载一致）：读取偏好 → 作为 `source="profile"` 初始假设传入 ForgeWorkbench。
- **不在 `/api/forge` 内重复注入**：否则用户在 Forge 内 dismiss 掉某条 profile 假设后，服务端会因「无法区分『被删除』与『客户端未传』」而把它重新加回——这是真实 bug。改由 Server Component 注入种子 + 客户端按既有 dismiss/edit 流程提交，dismiss 自然生效。
- **back-compat**：无偏好 → 种子为空 → `/forge` 与 `/api/forge` 行为与之前完全一致（旧请求兼容）。

## 鉴权 / 错误码（对齐现有风格）

- 所有偏好 API 必须登录：未登录 → `AUTH_REQUIRED`（401）。
- 参数不合法 / 非法 JSON / 三字段 trim 后为空 / 非 canonical intent → `VALIDATION_FAILED`（400）。
- 单条 PUT/DELETE 的他人 / 不存在 / 非法 id → `PREFERENCE_NOT_FOUND`（404，不泄露存在性）。
- DB 异常 → `DATABASE_ERROR`（500）。鉴权顺序：解析 body →（写）鉴权 → 校验 → RLS 读写。
