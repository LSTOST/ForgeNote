> 票：受保护配方详情页展示当前用户可见 recipe 的完整字段，并支持沿用该配方「换输入重跑」生成一个新的 Forge session。
> **状态：Done。** 自动验证、匿名权限边界、真实 Chrome 登录态详情/重跑闭环均已通过。

## 范围内

- 新增受保护 `/recipes/[id]` 详情页（Server Component 鉴权）。
- 详情展示：`name`、`intent_type`、`created_at`、`updated_at`、`usage_count`、`source_session_id`、核心 `fields`（受众/目标/语气/视觉规范）、`structure`、`variables`、`negative_rules`、`acceptance`、`schema` 摘要。

## 实现要点

- 详情页鉴权在 Server Component 顶层执行：未登录 / Supabase 未配置 → `redirect("/login")`。
- 非法 UUID / 读取失败 / 他人 / 不存在 / 已删除 recipe → 统一渲染「配方不存在或不可见」态，**不泄露存在性**（不区分"不存在"与"无权限"）。
- recipe 读取统一过滤 `deleted_at is null`，依赖 RLS（`auth.uid() = user_id`）。
- `POST /api/recipes/:id/rerun` 校验顺序（与 `/api/forge` 一致）：解析 JSON → body schema → **鉴权** → 路径 UUID → 输入空/超长 → RLS 读 recipe。
  - body schema（要求 `rawInput`）在鉴权前，故匿名缺字段为 `VALIDATION_FAILED`（与 `/api/forge` 同）。
- 沿用配方：从 `recipe.fields` 派生 `source="recipe"` 的假设条（受众/目标/语气/视觉风格/结构），与 `intent_type` 一并喂入 `generateContentPackage`，使新输入沿用该配方。
- 新 session：`source_recipe_id = recipe.id`，`status=completed`；失败落 D-04 草稿（`status=draft`、`outcome=null`、`error_code/message`，仍记 `source_recipe_id`）。
- 原 recipe 不被覆盖；成功后 `usage_count + 1`（基于已读取值自增；写失败不回滚已成功的 session）。
- 错误码对齐 API-CONTRACT §3 / §5.8：`AUTH_REQUIRED` 401 / `RECIPE_NOT_FOUND` 404 / `INPUT_EMPTY` 400 / `INPUT_TOO_LONG` 400 / `VALIDATION_FAILED` 400 / `GENERATION_FAILED` 500 / `MODEL_NOT_CONFIGURED` 503 / `DATABASE_ERROR` 500。
- 成功后跳 `/forge?session=<新 sessionId>`：`/forge` 按 RLS 预载该 session，completed → 成功态呈现，draft/失败 → 回填输入 + 错误态。不绕过 RLS、不使用 service role。

## 残余风险

- 新 session 的 `source_recipe_id` 指向 recipe id：经**代码路径**确认（rerun 成功与草稿两条 INSERT 均写入 `source_recipe_id = recipe.id`），并有强间接证据——重跑沿用了该配方的派生假设、且**正是该 recipe id**的 `usage_count` 由 0→1、新 session 可被当前用户读取并渲染。未做直连 DB 的 `source_recipe_id` 字段读取：浏览器自动化下直接复用登录态 token 查询生产 Supabase REST 被安全策略拦截（属未授权的凭据复用 + 直连后端），未绕过。与 I-08 一致按代码路径核对并记录为残余风险，不阻断 I-10 Done。
