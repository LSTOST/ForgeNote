# ForgeNote UX Writing 文案规范

> 版本：v1.1（2026-07-12 增补 §13 中文排版规则）  
> 目标执行者：Codex  
> 使用范围：ForgeNote 用户界面文案、按钮文案、状态文案、导航文案、页面说明文案  
> 配套文档：`docs/UI-DESIGN-SYSTEM-REFACTOR.md`、`docs/PRD-M2.md`、`docs/UIUX-M2.md`

---

## 0. 执行结论

ForgeNote 当前界面存在大量内部术语直接暴露给用户的问题，例如：

- 配方
- 派生
- 雷达
- 账号大脑
- 结构稳定
- 待裁决
- slot
- strategy
- blocker
- context.granularity

这些词可以继续存在于代码、schema、PRD、技术文档中，但不应该作为用户界面主文案。

本规范的目标是：

> 把 ForgeNote 的界面文案从“系统内部语言”改成“用户动作语言”。

用户看到界面时，必须立刻知道：

1. 这个功能能帮我做什么；
2. 点这个按钮后会得到什么；
3. 当前状态是否可以继续；
4. 下一步该做什么。

---

## 1. 文案总原则

### 1.1 用户语言优先

界面文案必须描述用户能理解的动作或结果，不描述系统内部模块。

错误：

```text
派生到小红书
保存为配方
选题雷达
结构稳定
```

正确：

```text
生成小红书版本
保存这套写法
找下条选题
内容框架已确认
```

---

### 1.2 按钮必须说明“点击后得到什么”

按钮文案优先使用动词开头。

推荐动词：

```text
分析
生成
保存
继续
复制
修改
补充
选择
查看
```

避免动词：

```text
派生
裁决
渲染
执行
应用配置
```

示例：

| 不推荐 | 推荐 |
|---|---|
| 派生到小红书 | 生成小红书版本 |
| 保存为配方 | 保存这套写法 |
| Resolve Decision | 确认这个选择 |
| Generate Structure | 生成内容框架 |
| Run | 开始生成 |

---

### 1.3 状态文案必须告诉用户“能不能继续”

状态文案不要展示技术状态，要展示下一步。

| 内部状态 | 不推荐 | 推荐 |
|---|---|---|
| stable | 结构稳定 | 内容框架已确认 |
| unstable | 结构不稳定 | 还差一步 |
| blocker | 稳定性阻碍 | 需要确认 |
| pending | 待裁决 | 待确认 |
| draft | 草稿态 | 还可以继续修改 |
| completed | 已完成 | 可以生成平台版本 |

---

### 1.4 代码可以保留内部词，UI 不可以

可以这样分层：

| 层级 | 可用词 |
|---|---|
| 代码 / Schema | recipe、derive、radar、structure、slot、strategy |
| PRD / 技术文档 | 配方、派生、雷达、结构稳定 |
| 用户界面 | 写法、生成平台版本、找选题、内容框架、表达方式 |

示例：

```text
代码中：saveRecipe()
界面上：保存这套写法
```

```text
代码中：deriveToPlatform()
界面上：生成小红书版本
```

## 2. 禁用词表

以下词汇禁止作为用户界面主文案出现。

| 禁用词 | 原因 | 推荐替代 |
|---|---|---|
| 配方 | 抽象，用户不知道是模板、结构还是规则 | 写法 / 这套写法 |
| 派生 | 技术词，用户不会这样表达 | 生成平台版本 |
| 雷达 | 模块感强，不够直白 | 找选题 / 本周可写选题 |
| 账号大脑 | 偏中二，有黑箱感 | 账号分析 |
| 结构稳定 | 系统状态词 | 内容框架已确认 |
| 结构不稳定 | 系统状态词 | 还差一步 |
| 待裁决 | 翻译腔，像系统任务 | 待确认 |
| slot | 技术字段 | 段落 / 内容环节 |
| strategy | 技术字段 | 写法 / 表达方式 |
| blocker | 技术字段 | 需要确认 / 还差一步 |
| context.granularity | 内部字段名 | 内容详细程度 |
| modality | 技术词 | 表达形式 |
| renderer | 技术词 | 生成方式 |
| prototype | 技术词 | 内容类型 |
| re-run | 技术词 | 再写一条类似的 |

## 4. 页面级文案规范

---

# 4.1 First-run / 首次登录页

## 无账号分析状态

页面标题：

```text
先分析你的账号
```

副标题：

```text
粘贴账号简介和近期内容，ForgeNote 会判断你适合写什么、哪些表达有效、下一条内容该怎么做。
```

输入框 placeholder：

```text
粘贴你的账号简介、定位、近期 3-5 条内容，或你想继续优化的账号方向……
```

主按钮：

```text
分析我的账号
```

次按钮：

```text
跳过，直接写一条内容
```

三张说明卡：

```text
账号定位
识别你的受众、主题边界和表达风格。

有效写法
总结哪些内容更值得继续，哪些表达容易跑偏。

下次建议
给出选题、结构和平台方向，不再从空白开始。
```

禁用：

```text
先建立你的账号大脑
生成账号大脑
账号大脑已建立
```

---

副标题：

基于你的账号分析，继续上次内容，或从本周可写选题开始。

## 中区 Stage A：内容框架

标题：

```text
内容框架
```

说明：

```text
这是这条内容的写作顺序。确认后，系统会生成可编辑正文。
```

结构块格式：

```text
钩子
问题开头

一个人住，厨房失控往往不是不会做饭，而是没有固定补货节奏。

为什么这样写：
从真实生活阻力切入，更容易让独居用户代入。
```

主按钮：

```text
生成正文
```

状态：

```text
内容框架已确认
还差一步
需要确认
```

禁用：

```text
结构稳定
结构不稳定
Stability blocker
待裁决
```

## 5. 按钮文案规则

### 5.1 主按钮

主按钮必须清楚表达下一步结果。

推荐：

```text
分析我的账号
生成内容框架
生成正文
生成小红书版本
保存这套写法
继续编辑
复制结果
```

不推荐：

```text
开始
提交
运行
派生
保存配方
应用
执行
```

---

### 5.2 次按钮

次按钮可以表达跳过、取消、返回，但要具体。

推荐：

```text
跳过，直接写一条内容
返回修改内容框架
继续编辑正文
稍后再补充
```

不推荐：

```text
跳过
返回
取消
下一步
```

说明：

`下一步` 只有在流程极清楚时可以使用，否则应写明下一步是什么。

## 6. 状态文案规则

状态文案要回答：当前能不能继续。

推荐：

```text
内容框架已确认
还差一步
需要确认内容详细程度
正文已生成，可以生成平台版本
平台版本已生成
账号分析已完成
账号资料还不够完整
```

不推荐：

```text
stable
unstable
blocker
pending
completed
ready
```

---

## 7. 空状态文案规则

空状态不要只说“暂无数据”，要告诉用户能做什么。

错误：

```text
暂无内容
```

正确：

```text
还没有内容。
先写一个想法，ForgeNote 会帮你生成内容框架。
```

错误：

```text
暂无账号大脑
```

正确：

```text
还没有账号分析。
粘贴账号简介和近期内容，系统会先判断你的受众、语气和有效写法。
```

## 9. Codex 执行要求

Codex 在执行 UI 重构时，必须同时执行文案清理。

### 9.1 扫描范围

扫描以下目录中的用户可见文案：

```text
src/app/**
src/components/**
src/lib/**
docs/UI-DESIGN-SYSTEM-REFACTOR.md
```

重点查找：

```text
配方
派生
雷达
账号大脑
结构稳定
结构不稳定
待裁决
slot
strategy
blocker
context.granularity
prototype
modality
renderer
```

### 9.2 替换要求

1. 用户界面文案必须按本规范替换；
2. 代码变量名可以保留内部词；
3. API 路径、schema、测试 fixture 可以保留内部词；
4. 如果内部词出现在用户界面，需要改成用户语言；
5. 如果不确定某个词是否用户可见，优先视为用户可见并改掉。

### 9.3 不要破坏业务逻辑

不要为了改文案而改：

```text
数据库字段
API contract
Zod schema
路由路径
测试 fixture 的结构字段
```

可以改：

```text
按钮文字
页面标题
说明文案
空状态文案
badge 文案
toast 文案
panel 标题
navigation label
```

## 10. 验收标准

完成后必须满足：

1. 用户界面不再出现“配方 / 派生 / 雷达 / 账号大脑 / slot / strategy / blocker / context.granularity”等词作为主文案；
2. 按钮文案变成用户动作，例如“生成正文”“生成小红书版本”“保存这套写法”；
3. 右栏全部是普通用户能理解的人话；

## 13. 中文排版与标点规则（v1.1 新增，依据 Fenng/tech-doc-style-chinese）

> 适用于全部 UI 文案、提示、说明、空状态。与前述规则冲突时，本节与 §1–§3 同级，就严不就宽。

### 13.1 标点

- 引号一律用直角引号「」，嵌套用『』；禁止弯引号 “ ” 与英文引号 " "。
- 中文语境用全角标点（，。：；？！）；不用中英文标点混排（禁止「中文句子, 英文逗号」）。
- 省略号用 ……，不用 ... 或 。。。
- 破折号少用；「不只是 X——更是 Y」句式属于 AI 味，禁止。

### 13.2 中英文与数字混排

- 中文与英文单词、数字之间加一个空格：「输出 3 个版本」「按 Enter 发送」。
- 全角标点与任何字符之间不加空格。
- 英文专名保持官方大小写：JavaScript、GitHub、OpenRouter；不用缩写变体（禁 JS 代替 JavaScript 出现在 UI）。

### 13.3 称呼与语气

- **UI 文案不直呼用户**：避免「你/您」开头的句式，改客观陈述。
  - ❌「你可以直接修改正文」 → ✅「正文可直接修改」
  - ❌「你还没有内容」 → ✅「还没有内容」
- 禁互联网黑话：赋能、抓手、闭环、打通、心智、颗粒度（UI 层；文档层已由 §2 约束）。
- 状态词不做机械直译：Success→「已完成」而非「成功」；Invalid→ 具体说什么不合法。

### 13.4 按钮补充规则（与 §5 合并生效）

- 按钮 = 下一步动作的结果，不复述页面标题。
- 一屏最多一个主按钮（brand 实心），与 `docs/DESIGN.md` §4 一致。
- 不在按钮文案里放 emoji 与感叹号。
