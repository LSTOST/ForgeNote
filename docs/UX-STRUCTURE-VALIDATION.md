# ForgeNote · UX 结构验证计划

> 状态：❄️ **已冻结（2026-07-12，G0S-12 阶段 0）——随 CCOS 冻结；可复用骨架将折入 PHASE-2**  
> 文档版本：v1.0  
> 日期：2026-07-11

## 1. 目的与范围

本计划用于验证 `USER-JOURNEY.md`、`INFORMATION-ARCHITECTURE.md` 和 `UX-FLOW.md` 中的关键假设，再决定低保真结构是否进入详细交互和 UI 设计。

本轮验证聚焦：

- 关键任务与实际创作习惯是否一致
- 信息架构和导航是否符合用户心智
- 页面内容层级是否支持快速决策
- 主流程、可选分支和异常恢复是否可理解
- “历史”“工作流”“版本”等核心概念是否清晰

本轮不验证品牌视觉、颜色、字体、动效、工程性能或 AI 文本质量上限。

## 4. 参与者

### 4.1 招募标准

必须满足：

- 独立经营至少一个内容账号
- 最近 30 天发布不少于 4 条内容
- 主要内容属于科技、AI、商业、投资或个人成长之一
- 日常使用至少两类工具完成创作，例如笔记工具、AI、翻译或平台草稿箱

优先满足其一：

- 同时经营国内和海外平台
- 有中英双语输出经历
- 每周固定进行内容策划
- 曾创建自己的提示词、模板或写作流程

排除：

- 主要由团队代写或运营，本人不参与流程
- 最近三个月没有持续创作
- 仅进行视频剪辑且没有文本策划过程

### 4.2 样本结构

目标 6–8 人：

| 类型 | 数量建议 |
|---|---|
| 单平台、单语言创作者 | 2 |
| 多平台、单语言创作者 | 2 |
| 多平台、中英双语创作者 | 2–4 |

## 5. 筛选问卷

1. 你目前运营哪些内容平台？
2. 过去 30 天大约发布了多少条内容？
3. 你的主要内容领域是什么？
4. 你是否会将同一内容改写后发布到其他平台？频率如何？
5. 你是否制作过中文和英文两个版本？频率如何？
6. 从灵感到发布，你通常使用哪些工具？
7. 创作过程中哪些环节由你本人完成？
8. 你是否保存过提示词、模板、文章结构或固定流程？
9. 是否愿意在访谈中展示一篇最近内容的真实创作过程和中间材料？

## 6. 研究方法与场次

### 6.1 第一部分：情境访谈，30–40 分钟

目标：了解真实行为，不让参与者先看 ForgeNote 方案。

顺序：

1. 介绍与授权
2. 回顾最近一篇真实内容
3. 展示工具链和中间材料
4. 追问中断、返工与版本管理
5. 了解跨语言、跨平台和复用行为

### 6.2 第二部分：结构化可用性测试，35–45 分钟

目标：验证低保真原型中的页面结构、流程与术语。

规则：

- 使用思考出声法
- 主持人只读任务，不提示点击位置
- 先观察，再追问
- AI 结果使用固定样例，避免内容质量干扰结构判断

### 6.3 收尾访谈，10 分钟

- 让用户复述 ForgeNote 的用途
- 询问最有价值和最困惑的部分
- 比较现有工具链
- 询问是否愿意在下周真实使用

## 7. 情境访谈提纲

### 7.1 最近一次真实任务

- 请选一篇最近发布的内容，从最初想法开始讲。
- 灵感最初出现在哪里？当时如何保存？
- 从保存到正式创作间隔多久？
- 开始创作时如何找回材料？
- 哪一步最耗时间？哪一步返工最多？
- 你如何判断内容已经“可以发布”？

### 7.2 AI 与结构

- 你通常在什么时候使用 AI？
- 你会先让 AI 出结构还是直接写正文？为什么？
- AI 理解错方向时你怎么纠正？
- 你会保留哪些版本？如何比较？

### 7.3 语言与平台

- 同一内容会发布到哪些平台？如何调整？
- 中文与英文版本是翻译还是重新写？
- 先决定语言还是先决定平台？
- 哪些平台规则最容易出错？

### 7.4 复用

- 下次做相似内容时会复用什么？
- 你把这些方法存在哪里？
- “工作流”“模板”“创作方法”分别让你想到什么？
- 你会在什么时刻保存一套有效方法？

## 8. 可用性测试任务

测试材料使用同一条中等完整度的示例灵感，参与者也可在不泄露隐私的情况下使用自己的材料。

### T-01 保存不完整灵感

**情境：** 你刚看到一篇文章，产生一个粗糙想法，但现在没有时间写完。请保存它，稍后再继续。

观察：

- 是否找到新建入口
- 是否理解“暂存”与“保存并整理”
- 是否担心输入不完整不能保存

成功：灵感被暂存，用户能说明稍后从哪里找回。

### T-02 从灵感生成结构

**情境：** 现在准备继续这条内容。系统判断的方向有一处不符合你的意图，请纠正并生成结构。

观察：

- 是否理解整理结果可编辑
- 是否能发现原始灵感
- 是否理解结构的用途

成功：修正方向并得到用户愿意继续使用的结构。

### T-03 修改结构并生成草稿

**情境：** 结构中的第二部分不合适，请修改后继续生成草稿，并精简其中一段。

观察：

- 是否能局部编辑
- 是否担心重生成覆盖修改
- 是否理解当前阶段和下一步

成功：完成结构修改和草稿局部优化。

### T-04 创建目标输出

**情境：** 请为一个海外平台准备英文版本；如果你认为不需要某个步骤，可以跳过。

观察：

- 先选语言还是平台
- 是否理解来源版本
- 是否认为所有阶段都必须完成

成功：创建并找到正确的目标版本，能说明其来源。

### T-05 处理中断与恢复

**情境：** 平台版本生成失败。请在不丢失已有内容的情况下继续完成任务。

观察：

- 是否能识别局部错误
- 是否知道重试或手动编辑
- 是否担心其他版本受影响

成功：原地恢复，已有内容保持可用。

### T-06 保存并复用方法

**情境：** 你认为这次结构和语气以后还会用。请保存，并用它开始一条新内容。

观察：

- 是否理解保存范围
- 对“工作流”术语的反应
- 是否知道复用会创建新任务

成功：保存方法并创建新任务，原任务不被修改。

### T-07 找回历史内容

**情境：** 请回到刚才完成的内容继续编辑；然后说明如果想以它为基础做一篇新内容会怎么操作。

观察：

- 是否选择历史而非工作流
- 是否区分继续与创建副本

成功：正确恢复原任务，并能指出复用途径。

### T-08 无权限 / 受限状态检查

**情境：** 当用户没有登录、没有完整账号大脑或没有目标平台设置时，观察是否能理解系统给出的受限提示并继续完成其他任务。

观察：

- 是否把受限状态理解成错误页
- 是否知道自己还能继续哪些操作
- 是否知道什么时候需要补充账号资料
- 是否知道“保存后再继续”还是“先补充后继续”

成功：用户能正确理解受限原因，且不会中断主任务路径。

## 9. 记录模板

### 9.1 单场记录

| 字段 | 记录内容 |
|---|---|
| 参与者编号 | 仅使用匿名编号 |
| 创作者类型 | 平台、语言、频率、领域 |
| 当前工具链 | 灵感、写作、翻译、发布 |
| 关键原话 | 逐字记录并标明任务 |
| 任务结果 | 成功 / 部分成功 / 失败 |
| 行为路径 | 实际点击和回退顺序 |
| 犹豫点 | 停顿超过 5 秒或主动询问 |
| 错误点 | 进入错误页面、误解操作结果 |
| 信任问题 | 内容安全、AI 控制、版本覆盖担忧 |
| 新发现 | 不在原假设中的行为或需求 |

### 9.2 问题严重度

| 等级 | 定义 |
|---|---|
| S0 | 不影响任务，仅偏好差异 |
| S1 | 轻微困惑，可自行恢复 |
| S2 | 明显延迟或走错路径，需要尝试恢复 |
| S3 | 无法完成关键任务或产生内容丢失风险 |

## 10. 指标与通过门槛

| 指标 | 门槛 |
|---|---|
| P0 任务独立完成率 | 每项至少 80% |
| 首次找到新建入口 | 5 秒内至少 80% |
| 理解当前阶段与下一步 | 至少 80% |
| 正确区分历史与工作流 | 至少 80% |
| 正确理解版本来源 | 至少 80% |
| 严重问题 | 不允许存在未处理的 S3 |
| 同一 S2 问题发生率 | 不超过 2/6；超过则必须迭代 |
| “工作流”概念复述 | 至少 70% 能说明保存的是方法 |

门槛用于结构判断，不用于证明市场需求。样本量较小时同时保留行为证据与原话。

## 11. 综合与决策方法

### 11.1 证据标记

- **已验证**：至少 3 名目标用户出现一致行为，且无强反例。
- **部分验证**：不同用户群表现不同，需要分群设计。
- **被否定**：多数用户行为与假设相反，或导致 S3 问题。
- **待验证**：证据不足，不做确定性结论。

### 11.2 优先级公式

问题优先级 = 严重度 × 发生频率 × 核心任务权重。

优先修复：

1. 内容丢失或覆盖风险
2. P0 任务无法完成
3. 导航与对象心智错误
4. 主操作与下一步不清晰
5. 效率和表达问题

### 11.3 结论回填

- 用户与任务证据 → `USER-JOURNEY.md`
- 对象、命名、导航和页面层级 → `INFORMATION-ARCHITECTURE.md`
- 分支、异常和恢复 → `UX-FLOW.md`
- 已拍板范围或原则 → `DECISIONS.md`
- 功能优先级变化 → `MVP-FEATURE-LIST.md`

## 12. 结构验证 Gate

进入高保真 UI 前必须满足：

- P0 任务均达到完成率门槛
- 没有未解决的 S3 问题
- 历史与工作流的心智边界明确
- 版本来源关系可被目标用户理解
- 可选步骤不会被误认为强制流程
- 上游修改和 AI 失败都有可理解的恢复路径
- 关键术语已经验证或替换
- IA 和 UX Flow 已按研究结果更新
