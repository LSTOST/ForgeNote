# DSN-01 Open Design Prompt v1

> 用途：粘贴到 Open Design 的 `web-prototype` skill，用于生成 DSN-01 onboarding-first UI + account-level assumptions 原型。
> 模型基线：OpenRouter BYOK + `deepseek/deepseek-chat`。不要把 Gemini 当阻塞项。
> 产物规则：Open Design 只产设计参考；HTML/CSS/JS 不直接进入 ForgeNote 源码。原型结果必须回流到本目录的 `handoff.md` 与 `codex-review.md` 后，才能拆 I-20。

## 设计提炼

### 原型目标

DSN-01 只验证一件事：新用户第一次进入 ForgeNote 时，能不能在极短路径里感到“它比空白 ChatGPT 更懂我的账号，而且我几乎不用解释”。

本票不是内容包重设计，也不是工作台全量实现。重点是把“假设条”从表单默认值升级为账号级判断：AI 先替用户定方向，用户只改不对的地方。

### 页面范围

- Onboarding-first app shell：顶栏、左导航、当前任务区、账号入口、自动保存状态。
- 首屏输入态：一句价值承诺 + 大输入框 + 一个主按钮。
- 理解中：轻量骨架，不跳页，不丢输入。
- 方案态：AI 已定方向，内联可改标签，依据可展开。
- 失败重试态：可恢复、可重试、不丢草稿。
- 已修改状态：显示已确认数量，未改按默认理解继续。
- 工作台终态示意：只做一张低保真暖色参照，不交互。

不做：

- 内容包结果区重设计。
- 图像/卡片渲染。
- 配方库、历史版本、表现回填的完整界面。
- 复杂账号建模、多账号、发布、日历、自动学习闭环。
- “改写已有笔记”入口。

### 关键用户路径

1. 新用户打开 ForgeNote。
2. 看见一个安静但真实的 app 外壳。
3. 在首屏输入一个没整理好的模糊想法。
4. 按 Enter 或点击“生成内容方案 →”。
5. 进入“理解中”骨架态。
6. 看到 ForgeNote 先替他定好的方向。
7. 用户只点开/修改 0-2 个不对的标签。
8. 未改项按默认理解继续。
9. 点击“生成内容方案 →”。
10. 本票在结果区前停止，只给结果区占位。

冷启动增强路径：

1. 用户不想只靠单条想法。
2. 点击输入框下方轻量链接“想更准？贴一条你发过的帖”。
3. 粘贴一条过往帖。
4. 方案态展示更笃定的账号级标签，并能展开看到依据。

### 验收标准

- 首屏只服务“模糊想法 → 内容方案”这一条主路径。
- 原型像真实 web app，不像静态 landing page 或满屏说明文。
- 持久 app 外壳存在：顶栏、左导航、自动保存、账号入口。
- 交互态存在：hover、focus、active、loading skeleton、乐观更新、平滑过渡。
- 键盘基础存在：Enter 前进，Esc 返回，focus ring，合理 tab 顺序。
- 假设条呈现为“方向判断”，不是问卷、表单或逐条审稿。
- 第一行先证明 AI 读懂了用户想法本身。
- 标签默认全部已定，用户只改错的。
- 每条关键假设在设计 handoff 里能映射到 `rationale` + `confidence`。
- 过往帖冷启动能映射到可选请求字段 `account_post`。
- 无账号上下文时，不把模型不确定包装成告警，只给轻量收益提示。
- 不暴露内部 taxonomy：不要出现“内容包 / 小红书笔记 / 卡片 Prompt / M1”等实现术语。
- 产物能支持后续写 `handoff.md`、`codex-review.md`，并拆 I-20。

## Open Design 复制用 Prompt

```text
为 ForgeNote 设计 DSN-01 高保真可点击 web app 原型。中文界面。目标不是 landing page，而是一个第一天就像真实产品的创作工作台新用户引导态。

产品背景：
ForgeNote 帮单人创作者把模糊想法变成可以直接开始做的图文内容方案。北极星是：第一次用就觉得“它已经比空白 ChatGPT 更懂我的账号，而且我几乎不用解释”。

本次只做 M1 主路径：
模糊想法 → AI 先定方向 → 用户只改不对的假设 → 生成内容方案。

视觉方向：
- 暖色文字编辑风，像 Granola / iA Writer / Bear 的创作工具气质。
- 纸感、留白、克制、安静、适合停下来写。
- 不要冷灰 SaaS 后台感，不要营销页，不要大面积紫蓝渐变，不要装饰光球。
- 不花哨，不堆卡片，不做图像编辑器。
- 中文，字体使用 Geist 或同级无衬线。

真实 app 外壳硬要求：
- 顶栏：当前任务名、已自动保存 hh:mm、账号入口。
- 左导航：对新人安静，大多为空态或轻量禁用，不要吓人。
- 主区：当前内容任务。
- 外壳从第一屏就存在，不要先做一个独立 landing hero。
- 做出 hover / focus / active / loading skeleton / optimistic update / 平滑过渡。
- 键盘：Enter 前进，Esc 返回，所有可点控件有 focus ring，tab 顺序合理。
- 自动保存草稿：输入后显示“已自动保存 hh:mm”。

首屏输入态：
- 首屏只保留一句价值承诺、一个大输入框、一个主按钮。
- 标题文案：把一个模糊想法，变成可以直接开始做的内容方案
- 输入框 placeholder 表达：想法没整理好也能直接丢进来。
- 主按钮：生成内容方案 →
- 输入框下方最多放一行很轻的灰色链接：想更准？贴一条你发过的帖
- 删除流程解释、示例任务、平台/风格说明、字数统计、已认识账号风格、也可以直接开始等所有干扰。

理解中状态：
- 用户提交后留在同一个 app shell。
- 输入不丢。
- 主区出现轻量骨架和短状态文案。
- 不要夸张进度条，不要长解释。

方案态核心：
- 标题用：我先帮你定了这次的方向，改不对的就行
- 不要用“我对这次内容的理解”，那像 AI 交作业。
- 第一行必须先讲实质，证明读懂了用户想法本身。
- 示例用户输入围绕“要不要存、存多少、放哪、何时能动”。
- 实质摘要写成：这次我会替你讲清：要不要存 · 存多少 · 放在哪 · 什么时候能动
- 再给 3 个真正影响生成的判断，做成内联可改标签：
  1. 受众
  2. 内容形式
  3. 表达角度
- 平台并入内容形式。
- 视觉不要出现在这一步，也不要和后续卡片重复。
- 默认全部已定。用户唯一动作是点标签“改”。
- 不要每项放“认可 / 调整 / 不对 / 帮我定”四个按钮。
- 标签点击后打开小型 popover 或 inline editor，用户改完后标签变为已确认。
- 显示轻量状态：已确认 n/3，未改按默认理解来。
- 底部只留一个前进按钮：生成内容方案 →
- 按钮下方小灰字：没改的我按默认理解来。
- 不要并列“跳过，直接生成”。

依据与置信度：
- 每条标签可展开“为什么这么判断”。
- 设计里要能表达每条假设都有 rationale 和 confidence。
- confidence 三类：high / inferred / unsure。
- 不要把 unsure 设计成橙色告警大卡。需要用户拿主意的项只做靠后的轻量提示。
- 提示文案用：这项你定一下会更贴你的账号
- 有账号上下文版本：标签更笃定，展开依据时引用过往帖的具体线索。
- 无账号上下文版本：仍先给默认方向，但在靠后位置提示“贴一条过往帖会更准”。

冷启动过往帖入口：
- 输入框下方的“想更准？贴一条你发过的帖”打开一个轻量区域。
- 用户可粘贴一条过往帖。
- 不要做复杂账号画像表单。
- 原型中展示：有过往帖时，方案态里的受众/表达角度更像账号级判断，并能展开看到依据。

失败重试态：
- 理解失败时不丢输入和过往帖。
- 提供重试按钮。
- 文案短，不甩锅，不展示技术错误。

结果区：
- 本票不设计内容包结果区。
- 只放一个简洁占位，表示下一步会生成内容方案。
- 不要设计卡片渲染、图片生成、发布日历、内容库。

工作台终态示意：
- 另加一张暖色低保真不可交互参照。
- 熟练用户布局：
  左 = 账号与内容资产
  中 = 当前内容任务与内容方案
  右 = 本次假设条 / 配方 / 生成控制
  底 = 历史版本与表现回填
- 标注或视觉表达：新人第一次只看到中区任务 + 右区方案；其余区随账号资产、历史、配方攒起来再逐块显形，不是一上来就满载。

必须产出的可点击状态：
1. 首屏输入
2. 已输入并自动保存
3. 贴过往帖展开态
4. 理解中骨架
5. 方案态：有账号上下文
6. 方案态：无账号上下文
7. 标签编辑 popover / inline editor
8. 已修改若干项：已确认 n/3，未改按默认理解来
9. 理解失败 / 可重试 / 不丢输入
10. 工作台终态示意

硬约束：
- 中文。
- 不暴露内部术语：不要出现“内容包 / 小红书笔记 / 卡片 Prompt / M1 / schema / prompt / RLS”等。
- 不做 Canva 式图像或卡片渲染。
- 不重设计结果区。
- 不暗示“改写已有笔记”入口。
- 不做复杂账号建模。
- 不做发布、日历、多账号、团队协作。
- Open Design 生成代码只作为设计参考，不需要符合 ForgeNote 源码结构。

交付时请附带：
- 可点击原型。
- 关键状态截图。
- 与现有 /forge 的差异点清单。
- 给实现的 handoff 要点：
  - 每条假设需要 rationale。
  - 每条假设需要 confidence: high / inferred / unsure。
  - 请求需要可选 account_post 字段，用于粘贴一条过往帖做冷启动。
  - 实现票必须预算 app 外壳、交互态、键盘、自动保存草稿，而不是只渲染方案态。
```

## 回流到仓库的产物要求

Open Design 生成后，把结果落到：

```text
docs/design/dsn-01-open-design/
```

最低要求：

- `prototype.html` 或 `screenshots/`：原型导出或关键截图。
- `handoff.md`：按 `handoff.template.md` 写状态、交互、数据锚点、差异点。
- `codex-review.md`：按 `codex-review.template.md` 写 Pass / Conditional Pass / Fail 和是否允许拆 I-20。

没有 `handoff.md` 和 `codex-review.md`，DSN-01 不算完成，不能拆 I-20。
