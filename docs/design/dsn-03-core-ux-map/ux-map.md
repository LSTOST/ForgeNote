# DSN-03 · UX Map — M1 核心路径母图

> ForgeNote M1 项目级体验母图。**先固定路径与状态，再拆实现票**。
> 北极星（`ForgeNote_修订版方向.md`）：**单人创作者第一次用 ForgeNote，就感到「它已经比空白的 ChatGPT 更懂我的账号，而且我几乎不用解释」。**
> 三支柱：① 假设条＝可见的账号级判断（第一次赢）；② 内容包＝可发级结构化初稿（产物值得留）；③ 配方＝可见地把好判断带到下次（M1 最小留存）。
> 本图覆盖 6 条路径（要求 ≥5）。每条路径标注：触发 → 步骤 → 首屏所见 → 关键判断 → 成功信号 → 与北极星关系。

---

## 全局体验姿态（约束所有路径）

- **页面是一个会长大的工作台，两个锚点形态**：引导态（新人，M1 主战场）与工作台态（熟练用户，终态），**渐进显形、连续生长，不跳变、不硬切两套页面**。
- M1 主路径只认**一条**：`模糊想法 → 内容`。「改写已有笔记」等是另一条入口，延后，示例文案不得暗示它存在。
- 任意时刻**一个主操作**（参考 Linear）；空区在内容出现前**不渲染**（新人不背空面板的认知税）。
- 用户可见文案**不外泄内部 taxonomy**（不出现「内容包 / 卡片 Prompt / schema / RLS / M1」）。

---

## Path 1 — First Entry（首次进入）

```text
匿名访客 → 登录/注册（邮箱密码 / Google）→ /forge → 立刻看懂「下一步做什么」
```

| 环节 | 设计决策 |
|---|---|
| 首屏所见 | `/login` 左＝品牌 + 吉祥物（内容创作演变小队），右＝单一表单；**第一印象传达「内容创作伙伴」，不是冷 SaaS 表单**。 |
| 登录/注册关系 | 同页切换（Sign In / Sign Up）；密码占位「至少 8 位」**仅注册态**出现；注册后 → 邮箱验证待确认态。 |
| 认证恢复在哪 | 「忘记密码？」在密码行右；重置为独立轻流程（请求 → 邮件已发送 → 设新密码 → 完成），**不塞进登录主表单**。 |
| 进入 /forge 后 | 落到**引导态**：一句价值问句 +大输入框 + 3 个示例任务 + 「我会替你补这些」标签预览。**首屏唯一任务＝开始第一条内容**。 |
| 不可干扰首生成 | 不显示内容库 / 配方库 / 历史 / 表现回填（新人都是空的）。吉祥物动效**只在 /login 与空态**出现，不进工作台。 |

- **成功信号**：用户登录后 30 秒内理解「在这里我要把一个想法变成可发的内容」，并开始输入。
- **北极星关系**：第一印象 + 引导态首屏共同兑现「几乎不用解释」。

## Path 2 — First Generation（第一次生成，支柱 1 主战场）

```text
登录用户 → 输入模糊想法 →（可选贴一条过往帖）→ 看见账号级假设 → 确认/改一项 → 生成内容方案
```

| 环节 | 设计决策 |
|---|---|
| 桌面工作台布局 | 北极星四区：**左**＝账号上下文/内容资产/已存上下文；**中**＝当前任务输入 + 内容方案；**右**＝假设条/生成控制/保存配方；**底**＝当前 session/复用/表现连续性。引导态下右/底/左多为空 → 暂不显形。 |
| 移动降级 | 单列：价值问句 → 输入 → 假设（折叠摘要）→ 生成 → 结果；**不把桌面四区硬塞小屏**。 |
| 当前任务层级 | 中区永远是「这一次的任务」；方向判断与配方保存**不并排挤进中区**。 |
| 假设条 affordance | **减字版**：可扫摘要/标签 + 置信度（确定/推断/拿不准）；依据「点开才看」（「我假设受众是 X，因为 Y」）；只对「拿不准」默认展开纠偏；编辑像**掌舵**（确认/改/否决），不是填表。改一项 → 计数 `1/3` + 该条转「确定」。 |
| 生成 加载/失败/重试 | 加载＝中区骨架 + 安静进度（不喧宾夺主）；失败＝**保留输入与假设**、给明确重试，不清空；失败若已落草稿 → 写 `?session=` 可刷新恢复。 |

- **成功信号**：假设条编辑/认可率显著高于现状 17%；用户感到「它已经懂我，我只是微调」。
- **北极星关系**：这是「第一次赢」的核心；假设条是差异化被感知的地方。

---

## Path 3 — Content Plan → Recipe（内容方案 → 配方，支柱 2/3 衔接）

```text
生成出内容方案 → 用户判断值不值得留 → 保存配方 → 看到配方详情入口
```

| 环节 | 设计决策 |
|---|---|
| 结果层级 | 按「发布正文 / 逐页卡片文案 / 配图方向 / 发布前检查」组织（**可发级**，不是一坨文字 + prompt）；session 等调试信息不作为主视觉。 |
| 保存价值时刻 | 保存区先讲「这次会复用哪些判断（受众/形式/角度/结构）」，让用户判断**值不值得留**，不是只给名字输入框。 |
| 成功态 | 保存成功 → 明确「查看配方」入口（进入复用路径），不只是「已保存」。 |
| 空/错误态 | 未生成时结果区是引导空态；保存失败保留草稿名与重试。 |

- **成功信号**：`recipe_save` 背后的「不值得存」变成「这个值得留下复用」。
- **北极星关系**：支柱 2「产物值得留」+ 支柱 3 的入口。

---

## Path 4 — Recipe Reuse（配方复用，支柱 3）

```text
配方详情 → 换输入 → 重跑 → 落回 /forge?session= → 看懂「这次复用了什么」
```

| 环节 | 设计决策 |
|---|---|
| 配方详情结构 | 来源 session / 核心字段（受众/目标/语气/结构…）/ 换输入重跑表单，三段清楚分区。 |
| 重跑输入位置 | 详情页内独立「换输入重跑」区，明说「沿用本配方、原配方不改动」。 |
| 回 Forge 连续性 | 重跑成功 → `/forge?session=<new>`，结果仍是 Path 3 的结构化层级；可见提示「这次复用了 X 配方的判断」。 |
| session 身份/恢复 | URL 即身份；刷新可恢复；新建/重新定方向清理旧 `?session=`，不把新任务误绑旧结果。 |

- **成功信号**：`recipe_rerun` 成立，用户感到「省了重复输入、上次的好判断被带过来了」。
- **北极星关系**：支柱 3，M1 只做「可见地带过来」最小形态。

---

## Path 5 — Return & Feedback（回访与表现回填，M1 最小）

```text
回访用户 → 找到既往配方/结果 → 记录发布表现 → 之后回看表现仍在
```

| 环节 | 设计决策 |
|---|---|
| 历史在哪（M1） | 配方库 `/recipes` 承载可复用资产；既往结果靠 `/forge?session=` 持久化回看；**M1 不建独立「历史版本」面板**（工作台底部区，逐块长出）。 |
| 配方/session/表现关系 | 配方＝可复用判断；session＝一次具体生成；表现＝挂在 session 上的手动回填（仅本人可见、不自动抓取）。 |
| 最小反馈 UI | OutcomePanel 内「记录发布表现」轻入口：点赞/收藏/评论/涨粉区间 + 一句复盘；保存后回看仍在。 |
| 明确延后 | 表现 → 学习闭环（识别模式/自动更新假设）**不在 M1**；自动抓取、看板、多账号延后。 |

- **成功信号**：`return_session_rate` / `performance_fill_rate` 有信号（持久化带来价值）。
- **北极星关系**：留存引擎的最小骨架，验证「持久化是否值得」。

---

## Path 6 — Auth Recovery（认证恢复，横切 Path 1）

```text
忘记密码 → 请求重置 → 重置邮件已发送 → 邮件回跳设新密码 → 完成 → 回登录
仅 Magic Link 老用户：经此流程设密码后改用密码登录（迁移路径须先于 Magic Link 下线可用）
```

- **成功信号**：非 Google 用户有不依赖每次开邮箱的登录路径；老用户可迁移。
- **北极星关系**：移除「每次登录都要开邮箱」的第一步摩擦，保住 Path 1 的转化。
- 详见 DSN-02（`docs/design/dsn-02-login-auth/`）；DSN-03 只把它纳入母图与状态机。

## 路径 × 支柱 × 指标 对照

| 路径 | 支柱 | 主要指标 |
|---|---|---|
| P1 首次进入 | — | activation_rate（前置） |
| P2 第一次生成 | 支柱 1 | assumption_edit_rate / activation_rate |
| P3 方案→配方 | 支柱 2/3 | recipe_save_rate |
| P4 配方复用 | 支柱 3 | recipe_rerun_rate |
| P5 回访/表现 | 留存最小 | return_session_rate / performance_fill_rate |
| P6 认证恢复 | —（保转化） | activation_rate（防流失） |

> 母图原则：任何新实现票必须引用上面某条 **path + 某个 state + 某条 design baseline 规则**，否则不允许动手（见 `implementation-slices.md` 与 `design-system-baseline.md`）。
