# ForgeNote 修订版方向（Codex）

> 本文修订并取代《ForgeNote_差异化与竞品优先级建议.md》中"M1 直接约束"一节的执行结论。
> 战略部分采纳原建议；执行部分按 2026-06-23 线上实地核查 + 生产指标证据做了关键修正。
> 角色：以 Codex（技术负责人 / 定义正确性）身份给出方向纲，供 Owner 拍板、Open Design / Claude Code 落地。

---

## 0. 一句话

**原建议选对了战场（护城河=过程层，不是产出介质），但默认了"我们已经站稳"——而事实是闭环只有骨架、没有价值。修订版方向：不堆功能、不去做视觉渲染、不急着拉测试用户；先把闭环里最差异化的一环（假设条）做成"第一次就让人觉得 AI 懂我的账号"，并让单次内容包扎实到能用。证据立住了，迁移/留存才谈得上验证。**

---

## 1. 采纳原建议什么（不重复造轮子）

以下原判断成立，直接继承：

- **护城河是过程层**：`模糊想法 → AI 补假设 → 用户纠偏 → 内容包+配方双产出 → 表现回填 → 下次更贴近账号`。不在"生成文案"上卷。
- **不做视觉生产**：渲染卡片/做图是 Canva 赛道，明确**不**作为 ForgeNote 方向。（这一条也**撤回** Codex 此前"路 A：去做真出图卡片"的临时倾向——那是错的。）
- **反面参照**：不做爆款改写器、批量发帖、纯 Prompt 模板库、营销团队中台。
- **M1 收敛为留存/迁移问题**，不是堆功能。
- **竞品方向判断**（Jasper=账号记忆、AuthoredUp=格式预览+反馈、Copy.ai/Notion/Narrato=流程与上下文）整体可参考。

## 2. 修正原建议什么（盲点，必须纠）

原文档把"M1 应保留假设条/内容包/配方/回填"写成"保留现状即可"。线上核查 + 指标证据表明，这是最大的误导：

1. **战略对 ≠ 执行到位。** 现状是**骨架**，每一环都没强到产生"哇"。指标在说话：
   - `assumption_edit ≈ 17%`：被原文称为"核心交互"的假设条，**大多数人根本不改**——没看懂或不觉得值得动。
   - `recipe_save ≈ 8%`：12 次生成只沉淀 1 个配方，**复用资产这条腿没立起来**。
2. **原文最值钱的"学习闭环"，恰恰是 M1 里最没做的。** `回填 → 识别有效模式 → 更新默认假设/配方` 在产品里是**空的**：I-12 只把表现存下来，没有任何"识别/更新"逻辑；现有"记住为偏好"是**手动**点★，不是从表现自动学习。**护城河那句话目前是 PPT，不是产品。**
3. **"内容包"按原文自己的定义都没交付。** 原文要内容包含"配图方向、发布素材"，现状产出是"标题/正文/卡片 Prompt/话题"一坨文字——薄、不够直接可用。问题不是"缺图"，是**过程层产物不够扎实**。
4. **冷启动被完全忽略。** 闭环价值是**后置**的（要用很多次记忆才显现），但创作者的迁移决策发生在**第 1 次**。原文通篇讲长期闭环，没讲"第一次怎么赢"。这是最大产品风险却只字未提。
5. **"深拆 5 个竞品"有变拖延的风险。** 在自己一个环节都没做好前做纸面调研，容易自我安慰。竞品研究应轻量并行，不作为前置 gate。

## 3. 修订后的北极星

> **一个单人创作者，第一次用 ForgeNote，就感到："它已经比空白的 ChatGPT 更懂我的账号/受众，而且我几乎不用解释。"**

这条北极星把验证重心从"长期迁移"前移到"**第一次的体感**"——因为第一次赢不了，就没有第二次，长期闭环无从谈起。

核心赌注：**差异化在第一次就要兑现，靠的是"假设条"——让用户看见 AI 的账号级判断、并低成本纠偏。** 配方/学习闭环是留存引擎，是第一次之后的事。

## 4. M1 重定义：三根支柱 + 明确延后

不再是"保留这些功能"，而是把闭环里**最差异化的环节做到真有价值**。按优先级：

**支柱 1（第一次赢）— 假设条：从"表单默认值"升级为"可见的账号级判断"。**
- 现状：平台/受众/语气等通用默认值，像填表。
- 目标：假设要**显得是从上下文推断来的、且足够准**；展示 AI 的依据（"我假设受众是 X，因为 Y"）；编辑像"掌舵"而非"填表"。
- 这是把 17% 编辑率背后的"没感觉"扭过来的关键。

**支柱 2（产物值得留）— 内容包：从"文字+提示词"升级为"能直接用的结构化初稿"。**
- 不做图；但内容包要扎实到创作者**几乎能直接发**：真正的正文、可选标题、按卡片拆好的**具体文案**（不是只给 prompt）、明确的配图方向。
- 让 `recipe_save 8%` 背后的"不值得存"变成"这个值得留下来复用"。

**支柱 3（留存，M1 只做最小）— 配方：只有当"复用确实让下一次更好"时才配存在。**
- 不扩功能；让"从配方重跑"**可见地把上次的好判断带过来**，用户能感到"省了重复输入"。

**明确延后（别在 M1 假装它存在）：**
- 表现→学习闭环（识别模式 / 自动更新假设）：真护城河，但属 M1 之后。
- 观测 SDK、runtime i18n、自动发布、内容日历、多账号、数据看板：全部延后。

## 5. 冷启动：第一次怎么赢（原文缺的部分）

- **首屏给一条 <1 分钟到"哇"的路径**：粘一个粗想法（或一篇过往帖）→ AI 立刻给出**账号级**假设 → 一份扎实内容包。"哇"必须发生在第一屏、第一次。
- **用"过往帖"冷启动账号画像**：让用户贴 1–2 条自己发过的内容，AI 据此推断受众/语气/风格，作为假设条的初始依据——这是把"账号记忆"在第一次就兑现的最便宜方式（对标 Jasper Brand Voice，但更轻）。
- **第一份产物即"可发"级**：第一次就让人觉得"这个我能直接用"，否则没有第二次。

## 5.5 页面形态：引导态 → 工作台（渐进显形）

ForgeNote 终态是"创作者的工作台"，但它不能一上来就以满载形态迎接新人——新人没有资产/历史/配方，满屏空区比文字墙更劝退。所以**一个会长大的工作台，两个锚点形态**：

**引导态（新用户，M1 主战场）**：首屏只做一件事——开始第一条内容任务。
- 顶部：一句明确价值说明（"你今天想做什么内容？"）。
- 中间：大输入框 + 3 个可点选示例任务。
- 输入前先亮出"我会替你补这些"：`[受众] [平台] [内容形式] [目标] [风格]`——第一屏就兑现"几乎不用解释"。
- 输入后 → 理解中 → "我对这次内容的理解"（假设条，账号级判断、**减字版**：可扫摘要/标签、推理点开才看、置信度导航、只对"拿不准"的展开纠偏）→ 生成内容包。
- **不显示**内容库 / 配方库 / 历史版本 / 表现回填——对新人都是空的，只增认知负担。

**工作台态（熟练用户，终态）**：
- 左：账号与内容资产；中：当前内容任务与内容包；右：假设条 / 配方 / 生成控制；底：历史版本与表现回填。

**渐进显形（关键，不要硬切两套页面）**：就一个工作台，每个区按"它装的东西出现了没"逐块出现——存了第一条账号记忆→左栏；有了第一个历史 session→底部；存了第一个配方→右栏。**连续生长，不跳变**（避免"我那个简单首屏怎么没了"的突兀）。

**M1 划线**：M1 ≈ 把**引导态**做扎实（支柱 1 在这里验证），它与现有"输入→假设→生成"结构接近；工作台是长大方向、逐块长出，不在 M1 一次盖完。

**单一主路径**：M1 只认"模糊想法 → 内容"。示例里"把已有笔记改写成更适合小红书"是**另一条入口**，另算、延后——示例不得暗示我们不会建的流程。

**参考拆解（只取一个点，别整体照搬）**：Granola/Notion=安静高密度的工作台结构（侧栏承载上下文、中间永远是任务）；Jasper=账号上下文资产化（压缩成 账号记忆→内容偏好→已验证配方→本次默认假设，不要企业化的十几个配置页）；Descript=同一空间完成、内容包局部可改（延后）；Linear=状态用轻标签不用大卡片、每页一个主操作；Canva=视觉资产/模板复用（延后，M1 只输出 视觉方向+图片 Prompt+版式建议+已验证视觉配方，不做可视化编辑器）。

## 6. 验证标准重定义

不再用"会不会迁移"（太远、太晚）。改用更早、更窄的信号，且对标 OPERATING-MODEL 的 Gate：

- **先验证支柱 1/2 的体感**，再谈留存：
  - 假设条参与度（编辑/认可率）显著高于现状 17%——证明差异化被感知。
  - 内容包"可用性"主观评分（首批 Owner+少量真实用户）——证明产物值得留。
- **这两点没立住前，不投入支柱 3 之外的任何东西，也不大规模拉用户**（拉来也只会重复"算不上 demo"）。

## 7. 竞品研究的正确用法

- **轻量、并行、有靶子**，不作为前置 gate。
- M1 阶段只精看两个、且只看一个点：
  - **Jasper**：Brand Voice / 账号记忆怎么在**第一次**就生效 → 喂支柱 1 与冷启动。
  - **AuthoredUp**：格式预览 + 贴近发布 → 喂支柱 2"可发级产物"。
- 其余（Narrato/Copy.ai/Notion/Canva…）后期路线图参照，不进当前。

## 8. 对当前在途的处置

- **V-01（拉真实用户验证）：挂起。** 前提（产品已到可被公平测试）不成立。先把支柱 1/2 做到有感觉，再小范围验证。
- **不走"路 A（视觉渲染）"。** 已撤回。
- **下一票从支柱 1/2 derive**，不是新功能堆叠，而是"把假设条做成账号级判断"+"把内容包做扎实"的产品/设计/prompt 工作——**本轮先用 Open Design 做 POC 原型，Claude Design 保底**（假设条交互、首屏冷启动体验），补上迄今最缺席的设计环节。Open Design 不直接吃 Claude Pro；若使用它生成，模型费用走 Owner BYOK/AMR。
- 顺带修执行潦草项（开发态 UI、内部 taxonomy 外泄如"内容包/小红书笔记/卡片 Prompt"、dev 口吻文案如"M1 不做自动发布"）——作为伴随打磨，不是主线。

## 9. 角色分工（落地）

- **Owner**：拍板北极星与支柱优先级；提供"对的参照"（你心里的标杆体验）。
- **Codex**：把支柱 1/2 拆成方向票与验收标准；定义"账号级判断/可发级产物"的可验收含义。
- **Open Design / Claude Design**：假设条交互 + 首屏冷启动体验的设计方向/原型（先于实现；Open Design 先 POC，Owner BYOK/AMR 付模型费；Claude Pro / Claude Design 保底）。
- **Claude Code**：按设计 + prompt 调整落地；地基（auth/生成链路/配方/RLS）复用，不推倒。
