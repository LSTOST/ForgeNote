# ForgeNote 重定义方向 v3.15（2026-07-02）

> 本文**取代**《ForgeNote_修订版方向.md》。这不是又一次修订，是 Owner 发起的推翻重来：
> 产品定位、目标市场、核心功能、UI 形态全部重定义。
> 触发原因：Owner 自己用下来体感不对（不如直接用 ChatGPT、产出不能直接发、流程像填表），且承认方向从未真正想清楚。
> 目标：认真做成**付费产品**。
>
> **v3.1 变更（Owner 反馈后）**：① 市场从"仅英文"改为**国内外双栖创作者**（一个账号大脑、多个平台适配层，MVP 锚定小红书 + X）；② UI 形态从卡片流改为 **IDE 式工作台**（左/中/右/下，参照 Codex 桌面版、Claude Code 桌面版、Cursor 的结合体）。
> **v3.2 变更（Owner 第二轮反馈）**：工作台四区按 Owner 的定义重写（§5）：对话是主驾驶、中间是单任务工作区、右侧是推理+控制、底部是内容时间轴；顶部任务栈、左下角设置（含 BYOK）、左右栏可收起；载体=先 Web 做出桌面感；模型=内置为主、BYOK 可选。
> **v3.3 变更**：视觉定稿——Owner 给出整套 Design Tokens（纸感 + 橙色仅语义强调，见 §5 末尾）。
> **v3.4 变更（Owner 收敛指令）**：从"信息型内容管理系统"收敛为**"单任务内容创作操作系统"**：中间唯一主舞台（一条内容的状态机），卡片流/列表/多任务并列移出主舞台；选题雷达改为左侧"新内容"触发的独立浮层；右侧收敛为轻量推理层（假设可编辑+依据默认折叠+单一控制动作，删除置信度/参数控制）；底部时间轴降级为抽屉式版本历史；顶部任务栈取消，任务切换回左侧"进行中"。
> **v3.5 变更（Owner 第一性原理重述）**：初衷排序定调——**首先服务 Owner 自己的内容创作，自己能用之后顺便成为商业项目**；人群不分国界（产品多语言）；内容形式以文字为主（可产出图片/视频的生成 prompt）；产出平台不设限（小红书/公众号/B站/X/Instagram/YouTube…）。"无国界、平台无关"是**架构原则**；MVP 落地顺序**跟随 Owner 真实使用**收敛（小红书 + X 起步）。

> 降级为渲染结果，不是一级概念。
> **v3.7 变更（Owner 最终收敛指令）**：产品定义统一为 **Content Structure Generation

> **v3.8 变更（Owner 再收敛）**：中间区**连"渲染结果合集"也禁止**——中区只允许一个对象
> （当前内容结构），状态机定名 **输入 → AI 结构化 → 结构确认 → 输出预览 → 局部编辑**；
> 输出预览一次只展示一个渲染，**多平台/多语言选择发生在结构之后的输出层**
> （预览块内的"输出为"选择器），不在输入决策层；左侧进一步去 CMS 化
> （配方库/账号记忆各收敛为单入口）。验收标准：用户进入系统只感知到
> **"我正在把一个想法逐步变成一条结构化内容"**。
> **v3.9 变更（Owner 强制收敛）**：确立铁律——**结构确认之前，任何完整内容不得渲染或
> 展示**。中区唯一对象是"当前结构的抽象表达"（inline 可改的结构节点），正文/卡片/渲染
> 一律不驻留；"生成预览"是**临时覆盖层**（关闭即消失）。右侧压缩为：结构字段 +
> 最多 1–2 个关键不确定点，其余默认折叠或删除。左侧降级为纯任务入口（新建 + 当前任务），
> 库/配方/账号记忆收进二级页面。底部弱化到仅版本号（点击回滚，不可浏览）。
> 感知目标：**"我在不断定义一条内容的结构"，而不是"我在看一篇生成好的文章并修改它"**。
> **v3.10 变更（Owner 强制性结构收敛重构）**：只允许删减与收敛，不允许新模块。
> 中区重写为**单结构槽位系统**——槽位只显示结构标签 + 短语级值（未展开的 slot），
> 不得出现完整句子/成品文本；**删除"确认结构"审批概念**（结构是持续编辑的状态流，

> （结构字段 + ≤2 个不确定点，操作按钮删除）；左侧仅两个入口（新建内容 + 当前任务，
> "库与记忆"二级入口也删除）；底部为纯版本回溯（回滚/派生，无内容摘要）。
> 唯一验收标准：**界面中任何部分让用户感知到"我在看一篇文章/卡片/成品"，即为设计失败。**
> **v3.11 变更（Owner 反内容化锁定，最终收敛）**：① **Recipe = Structure Schema**——
> 配方只保存槽位定义（slot schema），禁止保存任何内容文本或历史生成结果；
> ② 槽位**去语义化**：标签用纯结构节点（hook / context / insight / resolution），
> 值用结构 token（如 loss_open），禁止可读句式与内容语义短语——
> **结构不可直接阅读，只能被系统解释或渲染**；③ 预览保持惰性渲染，
> "结构未稳定时不可生成"；④ 右侧删除全部 AI 解释/推理/策略说明，
> 仅剩可编辑 slot 约束 + ≤2 个待裁决点。
> 最终验收：**系统中不存在任何"可被阅读的内容结构"；Recipe 只学习结构模式而非内容本身。**
> **v3.12 变更（Owner 修正"结构编译器"偏移）**：v3.11 把"可创作性"压扁了——纯 schema

## 0. 一句话

**ForgeNote 首先是 Owner 自己的内容创作系统——从"帮你把想法变成内容包"（成稿工具），转向"告诉你下一条该做什么、为什么会成"（选题引擎）+ 单任务创作操作系统。自己每周离不开它之后，它顺便成为一个商业项目：服务不分国界的文字型创作者（多语言产品），产出不限平台（小红书/公众号/B站/X/Instagram/YouTube…），以文字为主、可产出图片/视频的生成 prompt。核心承诺不变：打开产品 60 秒内，它对你账号的判断比空白的 ChatGPT 深一个量级。**

**初衷排序（第一性原理，v3.5 定调）：自用 → 可用 → 商业化。**
商业化是自用价值成立后的自然延伸（"顺便"），不是前置目标——
这决定了验证顺序、平台落地顺序、以及所有"要不要现在做"的判断标准。

## 1. 为什么推翻旧方向（不是修补）

旧方向（v2《修订版方向》）的三根支柱——假设条、内容包、配方——全部押在**成稿环节**：
你已经知道要做什么，产品帮你做得更快更好。推翻它的理由：

1. **成稿是 ChatGPT 的主场。** 通用对话模型写稿的体验和质量天花板极高，且每月都在提升。
   在这里卷，任何"流程化"产品都天然背负"不如直接聊"的重力——Owner 的体感（填表感、
   不如 ChatGPT）不是执行没做好，是**战场选错了**。
2. **真实的最痛点在上游。** 以 Owner 自己做号的经验，最耗心力、最愿意付钱解决的是
   **选题：不知道做什么能爆**。写本身反而是相对轻的。旧方向对这一段只字未提。
3. **选题恰恰是 ChatGPT 结构性做不好的。** 好的选题判断需要：你的账号历史和表现数据、
   你所在领域此刻正在发生什么、平台格式的胜率规律、以及随时间累积的"什么对你有效"。
   这些都是**数据和记忆问题，不是 prompt 问题**——空白对话框喂不进去，这才是能守住的护城河。
4. **旧方向最值钱的洞察可以转世。** "护城河=过程层、账号记忆、表现回填学习闭环"这些判断
   依然成立——但它们服务的对象从"成稿"换成"选题"后，价值兑现快得多：
   选题建议的好坏**每周都被真实数据检验**，学习闭环不再是 PPT。

**继承**：账号记忆、表现回填、学习闭环、"第一次就要赢"的冷启动思路、不做视觉生产。
**废弃**：以内容包为中心的产品结构、假设条（转世为账号画像）、配方体系。

## 2. 新定位（v3.5 按第一性原理重写）

### 第一用户：Owner 自己

产品的第一验收标准是：**Owner 每周做内容时真的离不开它**。
Owner 的真实使用 = 小红书「独居生活指南」运营中 + 英文 X 账号即将开——
这两个就是产品的第一批"真实账号"，一切功能按这两个账号的真实创作流打磨。
自己不用的功能不做；自己感觉不到价值的环节，不许拿"未来用户会需要"来辩护。

### 第二圈用户：不分国界的文字型单人创作者

自用价值成立后，商业化的对象是**任何语言、任何平台**的文字型单人创作者/founder-creator：
- 内容以文字为主（图文、thread、长文、脚本文案）；产品输出可附带
  **图片/视频的生成 prompt**（配图方向、封面构图、分镜提示），但不做媒体生成/编辑本身；
- 产品是**多语言的**（UI 多语言 + 内容语言不限），不预设"中文市场"或"英文市场"；
- 付费意愿有对标（海外：Hypefury $35/mo、Taplio $49/mo；国内：新红/千瓜年费数千元）。

### JTBD

> "别给我一百个泛泛的 idea。告诉我**下一条**该做什么、**为什么它会成**、**发在哪儿**——
> 而且这个判断是基于**我的账号**做出的，不是给所有人的通用建议。"

### 结构系统：3 种结构 × N 种渲染（v3.5 定稿，Owner 建模）

ForgeNote 不按"平台或内容形式数量"定义能力边界，而按**"内容表达模态 + 结构系统"**建模。
**系统本质是一个"内容结构生成引擎"，不是内容编辑器或多平台发布工具。**

三层管线：

- **输入层**：用户的模糊意图。
- **内容原型层（v3.15 补，用户语言的顶层词表）**：输入先归入 M1 五类原型之一——
  **经验复盘 / 知识解释 / 清单指南 / 观点表达 / 案例拆解**（超出范围明确提示）；
  表达模态由原型推荐，**用户不直接选择模态**。详见 PRD §3。
- **核心层（唯一关键产物）：Content Structure（内容结构）**，只有三类稳定结构：
  - **Narrative Structure（叙事结构）**——内容如何讲述（hook → context → insight →
    conclusion），覆盖所有线性文本表达；
  - **Visual Structure（视觉结构）**——信息如何分层与分页（封面 → 拆解 → 总结），
    覆盖图文/卡片/信息图类内容，本质是"文本 + 分镜 + 信息层级"；
  - **Temporal Structure（时间结构）**——信息如何在时间中展开（0–3 秒 hook → 冲突 →
    展开 → 收束），覆盖短/长视频与 AI 视频 prompt。
  - **叠加关系（精化）**：三种结构不是互斥分桶——叙事结构是基座，视觉/时间结构是
    叠加其上的排布层（图文 = 叙事 + 视觉分层；视频脚本 = 叙事 + 时间轴）。
    生成管线：`模糊想法 → 叙事结构（必有）→ 视觉/时间结构（按需叠加）→ 渲染`。
    复用发生在结构层：同一叙事结构今天渲染成小红书图文，以后叠加时间结构即成视频脚本。
- **输出层：Renderers（渲染器）**：X renderer、小红书 renderer、Instagram renderer、
  公众号 renderer、YouTube renderer、图像/视频 prompt renderer……
  平台不是内容类型，只是**同一结构的不同渲染器**；长度/调性/标签/发布惯例都是
  renderer 参数。**加一个新平台 = 加一个 renderer，不动结构，更不动大脑。**

账号大脑仍在这一切之上（平台/结构/语言无关），它决定的是**选什么题、用什么角度、
什么结构对你有效**；表现数据按 renderer 回填归因，喂给同一个大脑。

**UI 推论（v3.7 收紧）**：任何平台标签、内容类型切换都不是系统一级概念——
**平台切换 Tab 彻底删除**，输出区以"渲染结果堆叠"呈现同一 Content Structure 的
多个 render；结构沉淀为 **Recipe（结构资产）**，供下次复用（链路终点）。

**设计目标**：不是支持多少种内容形式，而是**稳定定义少量可复用结构，
以结构驱动多平台生成**。

**M1 划线（跟随 Owner 真实使用）**：结构做 Narrative + Visual 两类；
renderer 做三个：小红书、X、图像 prompt。Temporal Structure 与视频类 renderer
等真实需求出现再加（手艺与回填指标另成体系，离当前验证最远）。

推论：任何一个平台上验证过的题材规律，都可以成为另一个平台的选题依据
（标注"跨平台迁移，置信度降级"）——多平台创作者在任何单平台工具里都得不到这个。

**落地顺序跟随 Owner 真实使用（这是"不限平台"不变成广度陷阱的收敛机制）：**
起步 = 小红书 + X（Owner 在用的两个）；公众号/B站/Instagram/YouTube 等适配层，
等真实需求出现（自己开了号，或 design partner 强需求）再加，每次一个。

### 与竞品的关系（空隙在哪）

| 玩家 | 本质 | 对双栖创作者 |
|---|---|---|
| Typefully（$12.5–29/mo） | 写作环境 + 排程 | 英文平台 only，不懂你的账号 |
| Hypefury（$35/mo 起） | X 增长自动化 | "灵感面板"=爆款库，公认 generic；无中文平台 |
| Taplio（$49/mo 起） | LinkedIn 获客套件 | 面向 B2B 销售；无中文平台 |
| 国内数据工具（新红/千瓜等） | 平台数据监测 | 给你看别人的数据，不给你的账号做判断；无海外 |
| ChatGPT/Claude | 万能对话 | 能聊选题，但不知道你的账号和你的数据 |

所有海外工具的 ideation 都是**排程/写作套件上的附赠功能**，且都是"别人的爆款库"逻辑；
国内工具是数据面板逻辑。**没有人做"基于你自己账号的选题智能"，更没有人横跨中英文平台。**
这就是位置：不做排程（Typefully 们已做到位，做导出/集成即可），只做它们最弱的那一环，做深，
并用"跨平台账号大脑"筑第二道墙。

## 3. 核心产品循环

```
① 账号接入（每个平台：粘贴 profile + 最近 10–20 条帖子和表现）
        ↓
② 账号大脑（AI 对你账号的判断：受众/声音/已验证的题材与格式规律——可看、可纠偏；
   平台无关的部分共享，平台相关的部分分开挂）
        ↓
③ 选题推荐（每周 N 张"选题卡"，每张带依据：为什么是这个题、为什么是现在、为什么是你、
   建议发哪个平台（或双发）
        ↓
④ 展开成稿（选中一张 → 生成 Content Structure（叙事结构，按需叠加视觉/时间结构）→
   多 renderer 渲染（小红书/X/图像 prompt…）→ 局部编辑 → 复制/导出去发）
        ↓
⑤ 表现回填（发完把数据带回来，按平台归因，手动为主）
        ↓
⑥ 学习可见（"上周验证了 X、推翻了 Y"→ 更新账号大脑 → 下周选题更准；
   单平台学到的规律跨平台迁移，标注降级置信度）→ 回到 ③
```

差异化在 ③（选题卡的"依据"是别人给不了的），留存靠 ⑥（用得越久越准，且两个平台的
数据在同一个大脑里，迁移成本随时间翻倍上升）。成稿 ④ 保留但降级为"顺手的下游"。

## 4. MVP 划线

**做（且必须做扎实）：**
- 账号接入：MVP 用**粘贴**（profile 链接/文本 + 近期帖子和数据截图/文本），不依赖平台 API；
  小红书 + X 各一套接入引导；
- 账号大脑：结构化、可编辑、每条判断标注依据来源和适用平台；
- 选题卡：每周一批（先 5 张），每张 = 题目方向 + 钩子示例 + 建议平台与格式 + 依据 + 置信度；
  两个反馈动作："展开做这条" / "不是我的菜"（都是训练信号）；
- 展开成稿：brief + 编辑器，同一选题支持小红书版 / X 版两个标签页；产出以"能直接贴走"为标准；
- 表现回填：手动录入，按平台归因，回报**当场可见**（"这条数据推翻了我上周的判断 X"）；
- 多语言：产品 UI 中/英双语起步（设置里的语言版本），**内容语言不限**（大脑与生成天然多语言）；
- 图片/视频 prompt：内容包附带配图方向/封面构图/分镜提示的生成 prompt（文字产出，不做媒体生成）；
- 一套 eval：同一账号材料下，选题卡质量对比"喂给 ChatGPT"的盲测——生死线。

**不做（明确说出来，防止走形）：**
- 第三个平台（公众号/抖音/LinkedIn/YouTube 等，Gate 2 之前一律不加）；
- 排程与自动发布（导出/复制即可）；
- 平台 API 接入、实时趋势爬虫（不承诺我们供不起的数据）；
- 视觉生产、多账号（同平台）、团队协作、内容日历、独立数据看板。

## 5. UI 形态：三层系统的创作工作台（v3.13 定稿）

参照系：Codex 桌面版 + Claude Code 桌面版 + Cursor 的结合体。
载体：**先 Web，做出桌面感**（快捷键、命令面板、多栏），验证后再考虑桌面壳。
产品气质：创作者每天打开的工作站，不是网页工具。

**核心原则（v3.13）：ForgeNote 是三层系统，不是二维矩阵。**
- **意图层**：用户输入（模糊想法、纠偏指令）；
- **结构层**：Narrative / Visual / Temporal + hooks 等——只负责内容如何组织，
  **不允许出现任何平台概念**；
- **渲染层**：X / 小红书 / IG / YouTube / 图片 Prompt——只负责输出格式转换，
  **不允许改变结构逻辑**。

链路仍是 Intent → Structure → Render → Recipe，但**结构服务于内容生成，
不替代内容本身**：用户面对的是一份始终可读的内容，结构是它的操控面。
所有 UI 变化是状态变化而非页面跳转；所有操作围绕当前 active content；
**UI 中禁止出现结构 × 平台的二维交互模型**。

最终 UI 结构（概念稿见 `docs/design/redefine-2026-07/concept.html`）：
- **中 · 当前内容对象（唯一创作区，v3.13 定稿）**：**必须始终呈现"可阅读内容"，
  不能退化为 schema 或 token 列表**——中区是一份**平台无关的主稿**（可读、可局部编辑），
  结构服务于它的生成，不替代它本身。改右侧任何槽位或在对话框里说一句改法，
  主稿重新生成。对话输入框在本区底部。
  双层认知原则（v3.12）保留在数据层：机器侧仍是纯 schema（token），
  human_label 只做显示映射、不参与学习——但其呈现位置移到右栏结构控制。
- **左 · 内容流（可收起）**：意图层的入口——**新建内容、选题雷达（送牌）、
  进行中任务、历史**。左下角为帐户入口（头像，Codex 式菜单）。
- **右 · 结构控制（可收起）**：结构层的操控面——**slots**（human label 可编辑、
  hover 见 token）、**置信度**（该结构 × 该账号的历史胜率）、**应用中的规则**
  （来自账号记忆的约束）、**解释**（默认折叠的"为什么是这个结构"）。
  结构层禁止出现任何平台概念。
- **下 · 输出动作（渲染层）**：**平台只存在于这里**。布局：左侧=存为配方、
  版本（回滚/派生）；右侧=复制、**导出**（点击才弹出平台列表：小红书图文 /
  X thread / IG carousel / 图片 Prompt / YouTube 脚本…）。
  渲染层只做格式转换，不允许改变结构逻辑；
  **禁止结构 × 平台的二维交互模型**。（表现回填仍走发布后提醒，写回账号大脑。）

**参考拆解（v3.13 补，Owner 提供实测截图；只取机制，不取皮）**：**Laper（laper.ai，AI 剧本工作台）**
是目前最贴的 UI 对标——剧本即"结构渲染为可读内容"的百年范例，其工作台与三层系统同构。
**取**：① 冷启动首屏（问候 + 大输入框 + 意图 chips，点击在下方展开建议——ForgeNote 换成
"本周雷达"chip，点开即送牌）；② 结构元素与正文的内嵌绑定（编辑器知道每段是什么元素、
Tab 切换、ghost 占位）——落地为**槽位 ↔ 主稿段落联动高亮**；③ 空状态快捷键教学（Tab/Enter/@
ghost 提示）；④ AI 的"节奏医生"口吻（点评效果、留决定权，不代写）。
**不取**：左栏系统导航杂项（社区/回收站/推广卡）；右栏的"剧本格式"选择——格式属渲染层，
ForgeNote 必须沉底部（这正是 Laper 违反我们第 5 条的地方）；协作；分镜/节拍多视图。

**UI 验收标准（v3.13，按三层各一条）**：
**结构层**：不出现任何平台概念；schema 层不存内容文本，Recipe 只学习结构模式
（machine_key），human_label 不进入学习；
**内容层（中区）**：始终呈现可阅读内容，不退化为 schema 或 token 列表——
结构服务于内容生成，不替代内容本身；
**渲染层（底部）**：平台只在此出现，只做格式转换、不改结构逻辑；
UI 任何位置不得出现**结构 × 平台的二维交互模型**。

**Recipe = Structure Schema（v3.11 定稿；v3.15 修订呈现方式）**：配方**只保存内容生产方法**
（内容原型 + 结构槽位定义 + 常用受众 + 表达角度 + 语气 + 视觉规则 + 输出策略 +
已验证表现信号）——**禁止保存任何内容正文或历史生成结果**，否则系统退化为内容数据库。
Recipe 只学习结构模式，不学习内容本身；它既不是旧版的 prompt 模板，也不是内容快照。
**呈现方式（v3.15 裁决）**：配方是用户可见的显式资产（"我以后还想这样做"）——
底栏"保存为配方"+ 结构模板库入口；自动沉淀可作为补充，不再是唯一机制。

**渐进显形原则保留**：新用户首次进入，左右栏收起、时间轴只有一个节点，
中间就是一场对话：粘贴你的小红书/X profile → 60 秒后 AI 在对话里送上第一批选题牌，
每张带着"它凭什么这么判断"。**"哇"必须发生在这场开场对话里。**随资产出现，各区逐块长出。

**视觉基调（v3.3，Owner 定稿）**：设计 tokens 见
`docs/design/redefine-2026-07/design-tokens.md`，一句话定位——
**"一张可以执行 AI 生成的编辑纸"**：纸感底 + 极克制 UI + 橙色只做语义强调
（当前任务/关键改动/选中状态/风险提示，视觉占比 ≤3%）；禁止蓝色科技感、强渐变、重阴影；
UI 弱化、文字优先级最高，所有交互都"像写作动作"。支持深浅色切换（右上角），
左栏收起钮在左上、右栏收起钮在右上；左下角是**帐户入口（头像 + 名字 + 套餐，Codex 式）**，
点开才见：邮箱、个人资料、设置（语言版本 + AI 提供方/BYOK）、邀请好友、剩余用量、退出登录。

## 6. 核心风险（直说）

1. **广度陷阱（v3.5 后的最大执行风险）。** "不分国界、多语言、不限平台"作为架构原则成立，
   作为执行范围会让每个平台/语言都做得很浅。对冲：**落地顺序跟随 Owner 真实使用**
   （小红书 + X 起步，新平台等真实需求、每次一个）；架构上强制"大脑共享、适配层薄"；
   适配层做不深的部分（如小红书封面视觉）明确不做。
2. **自用样本的代表性。** "自己能用"消解了 founder-market fit 风险（第一用户就是 Owner），
   但也带来新偏差：自己的创作流不一定代表第二圈用户。对冲：商业化动作（Gate 1）依赖
   外部 design partner 验证，不从自用体感直接外推；X 侧 Owner 是新手，恰好可以验证冷启动。
3. **选题质量可能依然 generic。** 产品成立与否的赌注本身。对冲：eval 盲测作为 Gate 1 硬标准，
   赢不了 ChatGPT 基线就不往下走。
4. **数据供给受限。** 无 API、无爬虫时，"领域信号"先靠：用户自己粘贴的观察 +
   我们人工维护的少量领域动态（design partner 阶段可行），诚实标注每条依据的来源与新鲜度。
5. **回填闭环冷启动。** 手动回填是反人性动作。对冲：回填的回报当场兑现（学习可见），
   且第 1–2 周由我们替 design partner 手工回填，先证明"回填→更准"成立再谈产品化。

## 7. 验证 Gate（对齐 OPERATING-MODEL；v3.5 增加 Gate 0）

- **Gate 0（自用，一切的前提）**：Owner 连续 4 周用 ForgeNote 完成自己的真实周更内容
  （小红书 + X），且主观上**不想回到"ChatGPT 裸聊"**。这条不过，后面全部不谈——
  这就是"自己能用之后才顺便商业化"的操作化。
- **Gate 1（外部体感）**：10 个外部创作者（多平台/多语言，中文与非中文主场各 ≥3 人），
  首触 60 秒路径跑通后回答"这 5 张卡里有没有你**本周真的会做**的？"——≥6/10 人至少选中 1 张；
  盲测中选题卡质量胜过 ChatGPT 基线（同素材）。
- **Gate 2（留存信号）**：design partner 第 2、3、4 周是否**主动回来要下一批牌**（不催）；
  跨平台协同是否被感知（"A 平台的数据让 B 平台的选题变准了"至少被 2 人主动提及）。
- Gate 1 未过之前：不加新平台适配层、不做排程集成、不投任何增长。
